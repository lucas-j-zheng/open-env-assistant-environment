# Personal Assistant Env Behavior Spec

This document describes how the calendar environment works end-to-end: state, actions, tools, interrupts, negotiation, inbox, personal events, Markov observations, completion flags, reward, and key edge behaviors.

It is based on:
- `personal_assistant/server/personal_assistant_environment.py`
- `personal_assistant/server/seed_generator.py`
- `personal_assistant/models.py`
- `personal_assistant/server/app.py`
- `personal_assistant/openenv.yaml`

## 1) Environment Identity

- Name: `personal_assistant`
- Runtime: FastAPI (`openenv.yaml`)
- App entry: `server.app:app`
- Default port: `8000`
- Web interface app is created with `max_concurrent_envs=4`.
- Concurrent sessions are supported (`SUPPORTS_CONCURRENT_SESSIONS = True`).

## 2) Action, Observation, State Schemas

## Action

Type: `CalendarAction`

Field:
- `instruction: str`

Expected usage is a JSON string that contains a tool call:

```json
{"tool": "create_event", "args": {"title": "Meeting", "date": "today", "start_time": "10:00"}}
```

If `instruction` is not valid JSON tool format, the env returns a help/error-style message and does not dispatch a tool.

## Observation

Type: `CalendarObservation`

### Core fields
- `output: str` — tool result text (and possible interrupt message appended)
- `pending_tasks: int`
- `events_today: int`
- `flags_found: list[str]`
- inherited fields from `Observation`: includes `reward`, `done`, and `metadata`

### Markov state snapshot fields
These fields make the observation self-contained — an agent can act optimally from any single observation without needing conversation history.

- `state_summary: str` — compact text rendering of full state (for sliding-window prompts, ~500-1200 tokens)
- `calendar_snapshot: list[dict]` — all current calendar events (excluding cancelled)
- `discovered_preferences: dict[str, dict]` — contact preferences the agent has queried so far
- `discovered_constraints: list[dict]` — private constraints revealed via `get_contact_preferences`
- `constraint_status: dict` — current hard/soft violation counts and details
- `active_negotiations: dict[str, dict]` — in-progress negotiations (`{round, attempts, last_feedback}`)
- `resolved_negotiations: dict[str, bool]` — completed negotiations (success/failure)
- `unhandled_interrupts: list[str]` — interrupt messages not yet resolved (auto-cleared when corresponding flag is earned)
- `notifications_sent: list[dict]` — all notifications sent
- `inbox_snapshot: list[dict]` — inbox messages visible at current step
- `step_count: int` — current step number

Current metadata keys:
- `seed`
- `episode_today`

## State

Type: `CalendarState`

Fields:
- inherited `State` fields (including `episode_id`, `step_count`)
- `tasks_completed: int`
- `total_tasks: int` (18 in current config)

## 3) Episode Lifecycle

## Reset Flow

`reset(seed=None, episode_id=None)` does the following:
1. Clears progress and transient state:
- `_found`, `_notifications`, `_fired_interrupts`, `_pending_interrupt_msg`
- `_active_negotiations`, `_negotiation_resolved`
- `_discovered_preferences`, `_discovered_constraints`, `_negotiation_feedback`
- `_unhandled_interrupts`, `_inbox`
2. Resets attendee schedules to defaults (deep copy)
3. Deactivates description policy
4. Seeds deterministic episode context:
- Re-seeds RNG
- Picks `episode_today` from weekdays only (Mon-Fri) over 3 weeks from base date `2026-01-05`
5. Calls `_seed_initial_events` which uses `generate_episode_config(rng, today)` to produce:
- Initial work events (with an intentional conflict between two of them)
- Personal/immovable events (from seed generator)
- Initial inbox messages (scheduling requests, diplomatic emails)
- Dynamic interrupt steps (randomized per seed)
- Lunch-exempt event IDs
6. Marks initial conflicts as present
7. Recomputes completion flags and refreshes unhandled interrupts
8. Returns initial Markov observation

Initial events are generated by the seed generator and vary per seed. They always include:
- A morning meeting with Alice
- A lunch meeting with Dave (lunch-exempt)
- A cancellable meeting with Alice, Bob, Charlie (intentional overlap)
- A design/review meeting with Eve (overlaps with above)
- Personal events (e.g., gym, dinner) that are immovable

## Step Flow

`step(action)` executes in this order:
1. `step_count += 1`
2. Clear `_pending_interrupt_msg`
3. Process any scheduled interrupt for this step
4. Parse `action.instruction` as JSON and dispatch tool if valid
5. Recompute all completion flags
6. Refresh unhandled interrupts (remove those whose flags are now earned)
7. Append pending interrupt message (if any) to output
8. Build and return Markov observation via `_build_observation(output)`

## 4) Time and Date Resolution

Supported date inputs in tools:
- `"today"`
- `"tomorrow"`
- `"next <weekday>"` (e.g. `next tuesday`)
- `"<weekday>"` (e.g. `friday`, interpreted as next occurrence; same-day rolls to next week)
- explicit `YYYY-MM-DD` (passes through unchanged)

Important: `today` is episode-relative and deterministic per seed, not wall-clock system date.

## 5) Constraints Model

Constraints are evaluated from all current events. Some constraints are conditionally active.

Hard constraints:
- `no_meetings_during_lunch` (12:00-13:00 overlap forbidden, except exempt event IDs)
- `bob_no_mondays`
- `eve_not_before_10`
- `long_meetings_need_description` (only active after `description_policy` is activated by interrupt; personal events exempt)

Soft constraints:
- `max_3_meetings_per_day`
- `alice_prefers_mornings` (< 12:00)
- `charlie_prefers_afternoons` (> 13:00)

Constraint visibility:
- **Public** (`get_constraints`): `no_meetings_during_lunch`, `max_3_meetings_per_day`, `long_meetings_need_description` (when active)
- **Private** (`get_contact_preferences(person)`): `bob_no_mondays`, `eve_not_before_10`, `alice_prefers_mornings`, `charlie_prefers_afternoons`

`_is_constraint_active()` gates the `long_meetings_need_description` constraint behind the `description_policy` flag.

## 6) Available Tools (15 Total)

All tools are invoked via `instruction` JSON with `"tool"` and `"args"`.

## Read/inspection tools

- `list_events(date="today")`
  - Returns events for one day, tagged `[WORK]` or `[PERSONAL]`.

- `find_free_slots(date="today", duration_minutes=60)`
  - Computes free windows in 08:00-18:00 from calendar events on that date.

- `check_conflicts(date="today")`
  - Detects overlapping adjacent events by start-time ordering.

- `check_availability(person, date="today")`
  - Shows external commitments (static attendee schedule, may change via availability_change interrupt), calendar events, and merged free windows.

- `get_constraints()`
  - Shows public hard/soft constraints only. Hints that private constraints exist.

- `get_contact_preferences(person)`
  - Shows role, notification preference, notes, and private constraints for that person.
  - **Side effect**: records discovery in `_discovered_preferences` and `_discovered_constraints` for Markov state tracking.

- `check_constraint_violations()`
  - Returns counts and details for hard/soft violations.

- `read_inbox(status="all")`
  - Lists inbox messages filtered by `all`, `unread`, or `unreplied`.
  - Marks messages as read when viewed.
  - Only shows messages whose `received_at_step` <= current step.

- `check_personal_calendar()`
  - Shows personal (immovable) events.

## Mutating tools

- `create_event(title, date, start_time, duration_minutes=60, attendees="", description="")`
  - Builds event and may trigger negotiation gate.
  - On negotiation reject/fail: event is not created; feedback stored in `_negotiation_feedback`.
  - On success: event appended; returns creation message plus constraint warnings tied to that event title.

- `delete_event(title)`
  - Deletes all events whose title matches case-insensitively.
  - **Cannot delete immovable/personal events.**

- `edit_event(title, new_title="", new_date="", new_start_time="", new_duration_minutes=0, new_attendees="", new_description="")`
  - Partially updates first title match.
  - Returns change summary plus constraint warnings for updated event title.
  - Does not run negotiation.
  - **Cannot edit immovable/personal events.**

- `resolve_conflict(event_title, new_start_time)`
  - Moves first title match to new start time, preserving duration.
  - Returns warnings if constraints violated.
  - **Cannot move immovable/personal events.**

- `send_notification(to, message)`
  - Appends a notification record.

- `reply_message(message_id, reply)`
  - Replies to an inbox message.
  - Minimum 20 characters.
  - Must contain at least one keyword from `reply_keywords` to address the sender's concern.
  - Message must be received (step check).

## Tool dispatch behavior

- Unknown tool -> explicit "Unknown tool" response with available tool names.
- Bad args/signature mismatch -> explicit TypeError response from dispatch layer.
- Empty args -> treated as `{}`.

## 7) Interrupt System

Interrupts trigger by step number (after increment, before tool dispatch), once per episode. Step numbers are randomized per seed via `EpisodeConfig.interrupt_steps`.

Default interrupt step numbers (overridden per episode):

| Default Step | Type | Effect |
|---|---|---|
| 3 | `new_meeting` | Injects CEO Sync at 15:00-15:30 today |
| 5 | `inbox_update` | Adds contradicting follow-up message to inbox |
| 6 | `cancellation` | Marks lunch meeting as cancelled |
| 7 | `availability_change` | Extends Bob's Wednesday busy blocks (13:00-17:00) |
| 9 | `reschedule_request` | Asks to move morning meeting to 11:00 |
| 10 | `personal_event_change` | Changes personal event time, may create new conflicts |
| 12 | `policy_change` | Activates `long_meetings_need_description` constraint |

Interrupt handling:
- Each fired interrupt is recorded in `_unhandled_interrupts` with its message.
- `_refresh_unhandled_interrupts()` removes entries whose corresponding flags are earned.
- `_is_interrupt_handled()` maps interrupt types to their flag names.
- Interrupt message text is appended to the normal tool output for that step.
- Dynamic titles (morning meeting name, lunch meeting name) come from `EpisodeConfig`.

## 8) Negotiation System

Negotiation is only checked on `create_event`.

Trigger matching:
- Scenario must match title keyword and required attendee subset.
- Scenario is skipped only if already resolved successfully (`_negotiation_resolved[scenario] is True`).
- Failed scenarios (`False`) are retriggerable in the same episode.

Internal states:
- `_active_negotiations[scenario] = {"current_round": int, "attempts": int}`
- `_negotiation_resolved[scenario] = True|False`
- `_negotiation_feedback[scenario] = "last rejection message"` (for Markov state)

Round mechanics:
1. First matching create attempt always starts negotiation and is rejected with round 0 message.
2. Subsequent attempts increment `attempts`.
3. If `attempts > max_rounds`, negotiation fails (`False`) and active state is cleared.
4. If current round accepted, engine advances through remaining rounds:
- Conditional rounds can be skipped if condition does not apply.
- If any applicable round fails, returns that rejection and stores `current_round`.
5. If all applicable rounds pass, marks scenario resolved `True` and allows event creation.
6. On successful resolution, feedback is cleared from `_negotiation_feedback`.

Scenarios:

- `standup_negotiation`
  - Trigger: title contains `standup`, attendees include `Alice` and `Bob`
  - Max rounds: 3
  - Acceptance: `duration_minutes <= 20`
  - Extra behavior: from attempt 2 onward, failed responses include a hint about `get_contact_preferences`.

- `kickoff_negotiation`
  - Trigger: title contains `kickoff`, attendees include `Alice`, `Bob`, `Eve`
  - Max rounds: 4
  - Round 0 acceptance:
    - `duration_minutes <= 45`
    - Eve 30-minute buffer before and after (`_has_buffer`)
  - Round 1 conditional:
    - Applies only when `start_time >= 12:00`
    - Then requires `start_time < 12:00` (forces morning if afternoon proposal triggered this round)

## 9) Inbox System

The inbox holds messages from team members and external contacts. Messages have:
- `id`, `from`, `subject`, `body`, `type`, `core_ask`, `reply_keywords`
- `received_at_step` — message only visible after this step
- `read`, `replied`, `reply_text` — tracking fields

Initial messages are seeded by `EpisodeConfig.initial_messages`. A mid-episode contradicting message (`mid_episode_message`) arrives via the `inbox_update` interrupt.

Reply validation:
- Minimum 20 characters
- Must contain at least one keyword from `reply_keywords`
- Replies that don't address the sender's concern are rejected with guidance

## 10) Personal Events System

Personal events are immovable calendar entries (gym, dinner, etc.) generated by the seed generator.
- Tagged with `type: "personal"` and `immovable: True`
- Cannot be deleted, edited, or moved via any tool
- Tools return error messages like "Cannot delete/modify/move — this is a personal/immovable event"
- Exempt from the description policy constraint
- Work events must be scheduled around them
- Personal events can change mid-episode via the `personal_event_change` interrupt

## 11) Markov Observation System

Every observation includes a `state_summary` field — a compact text rendering (~500-1200 tokens) of the full environment state. This enables sliding-window agent architectures where only the system prompt + state summary + last N tool exchanges are kept in context.

The state summary includes:
- Step count and today's date
- Task completion progress (done/pending flags)
- Full calendar snapshot (all events, grouped by date, tagged WORK/PERSONAL)
- Known preferences (only people queried so far) + unqueried people list
- Current constraint violations (hard + soft)
- Active/resolved negotiation state with last feedback
- Unhandled interrupts (auto-cleared when resolved)
- Notifications sent
- Active policies
- Inbox status (message IDs, subjects, read/replied status)

Agent scripts use sliding window pattern:
```python
messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    {"role": "user", "content": latest_state_summary},
] + history[-SLIDING_WINDOW:]
```

## 12) Completion Flags (18 Total)

Flags are recomputed every step and can be revoked unless noted.

### Static flags (no interrupt prerequisite)

1. `standup_scheduled`
- Has event this week with:
  - title includes `standup`
  - attendees include Alice + Bob
  - duration <= 30
  - no attendee external schedule conflicts
  - no calendar overlaps on that day
- AND `standup_negotiation` must be resolved `True`

2. `focus_time_booked` (revocable)
- Event today, title includes `focus`, duration >= 60, within 08:00-18:00, no overlaps

3. `conflicts_resolved` (revocable)
- No overlaps today; only meaningful because initial conflict existed

4. `reminder_set` (revocable)
- Event with title including `dentist`, date in next-week window, afternoon (>=12:00), within working hours (end <=18:00)

5. `meeting_cancelled` (revocable)
- No event with title containing the cancellable meeting name
- At least one notification to Alice/Bob/Charlie mentioning old project or cancel

6. `kickoff_scheduled`
- Has kickoff event including Alice/Bob/Eve
- No hard-constraint violation for that kickoff title
- No attendee external schedule conflicts
- No calendar overlaps on that day
- AND `kickoff_negotiation` must be resolved `True`

7. `hard_constraints_clear` (revocable)
- All hard constraints satisfied globally

8. `preferences_optimized` (revocable)
- At least 2 soft constraints satisfied globally

9. `inbox_cleared` (revocable)
- All visible inbox messages have been replied to

10. `diplomatic_reply_sent`
- The message with `type: "diplomatic"` has been replied to

11. `work_life_conflicts_resolved` (revocable)
- No personal event overlaps any work event on the same date
- Only checked if personal events exist

### Dynamic flags (require interrupt to fire first)

12. `ceo_sync_accommodated` (revocable)
- CEO interrupt has fired
- CEO event at exactly 15:00-15:30 today with no overlap

13. `cancellation_handled` (revocable)
- Cancellation interrupt has fired
- No lunch meeting event remains

14. `reschedule_handled` (revocable)
- Reschedule interrupt has fired
- Morning meeting event exists at 11:00

15. `availability_drift_handled` (revocable)
- Availability change interrupt has fired
- No Bob events on Wednesday conflict with his new busy blocks

16. `client_request_updated`
- Inbox update interrupt has fired
- The contradicting message (`msg_contra`) has been replied to

17. `personal_update_handled` (revocable)
- Personal event change interrupt has fired
- The changed personal event has no overlap with any work event

18. `description_policy_met` (revocable)
- Policy change interrupt has activated the description policy
- All work events >30 minutes have a description (personal events exempt)

## 13) Reward and Done

At reset and after every step:

```text
reward = len(flags_found) / 18
done = (len(flags_found) == 18)
pending_tasks = 18 - len(flags_found)
state.tasks_completed = len(flags_found)
```

## 14) Seed Generator

`generate_episode_config(rng, today)` produces an `EpisodeConfig` dataclass containing:
- `initial_events` — varied meeting titles/times per seed
- `personal_events` — immovable personal events
- `initial_messages` — inbox messages (scheduling requests, diplomatic emails)
- `mid_episode_message` — contradicting follow-up message
- `personal_event_update` — new times for personal event change interrupt
- `interrupt_steps` — randomized step numbers for each interrupt type
- `lunch_exempt_ids` — event IDs exempt from lunch constraint
- `cancellable_title`, `lunch_meeting_title`, `morning_meeting_title` — dynamic titles

## 15) Important Behavioral Notes

- `edit_event` and `resolve_conflict` never trigger negotiation; only `create_event` does.
- Negotiation applies per scenario, not per event ID.
- Once a scenario is resolved `True`, future matching events skip negotiation in that episode.
- On cancellation interrupt, event is marked `cancelled=True` but still present until explicitly deleted.
- Lunch hard constraint exempts only seed-generated lunch event IDs; newly created lunch-overlapping meetings violate hard constraints.
- Title-based matching is used heavily (e.g., `"standup"`, `"kickoff"`, `"focus"`, `"dentist"`, `"ceo"`), so naming affects scoring.
- Personal events are immovable — all three mutating tools (delete, edit, resolve_conflict) check for `immovable` flag.
- Attendee schedules can change mid-episode via `availability_change` interrupt (deep copy from defaults on reset).
- The `long_meetings_need_description` constraint is inactive until the `policy_change` interrupt fires.
- Inbox messages have `received_at_step` gating — messages from future steps are not visible.
- Reply validation checks both minimum length (20 chars) and keyword relevance.
- Unhandled interrupts are automatically cleared from the state summary when their corresponding flags are earned.
- `_build_observation()` calls `_evaluate_constraints()` on every step (for both text summary and structured fields).

## 16) Minimal Action Examples

Read inbox:

```json
{"tool":"read_inbox","args":{}}
```

Check conflicts:

```json
{"tool":"check_conflicts","args":{"date":"today"}}
```

Create standup:

```json
{"tool":"create_event","args":{"title":"Team Standup","date":"friday","start_time":"10:30","duration_minutes":20,"attendees":"Alice,Bob"}}
```

Check a person:

```json
{"tool":"check_availability","args":{"person":"Eve","date":"tuesday"}}
```

Resolve overlap:

```json
{"tool":"resolve_conflict","args":{"event_title":"Design Review","new_start_time":"15:30"}}
```

Reply to message:

```json
{"tool":"reply_message","args":{"message_id":"msg_001","reply":"I'll schedule the standup for Tuesday at 11:00 AM with Alice and Bob."}}
```

Add description to existing meeting:

```json
{"tool":"edit_event","args":{"title":"Team Social","new_description":"Weekly team bonding and updates"}}
```

Check personal calendar:

```json
{"tool":"check_personal_calendar","args":{}}
```
